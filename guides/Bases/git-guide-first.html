Введение

Git - это система контроля версий, которая стала неотъемлемой частью современной разработки. Она записывает изменения в файлах проекта и позволяет в любой момент вернуться к любому сохранённому состоянию. Git работает локально на вашем компьютере и не требует постоянного подключения к интернету.

GitHub - это облачная платформа, которая хранит Git-репозитории и добавляет удобный веб-интерфейс, инструменты для совместной работы, отслеживания задач и автоматизации. Понимание разницы между Git (инструмент) и GitHub (сервис) - первый шаг к грамотному управлению проектами.

Этот гайд проведёт вас от установки Git и регистрации на GitHub до повседневных сценариев работы с ветками, коммитами и удалёнными репозиториями.

Установка Git и первичная настройка

Установка на Windows, macOS, Linux

Скачайте установщик с официального сайта git-scm.com/downloads. Для Windows выбирайте 64-битную версию, для macOS - установщик .dmg или через Homebrew, для Linux используйте пакетный менеджер (например, sudo apt install git на Ubuntu).

В процессе установки на Windows оставьте все параметры по умолчанию, но убедитесь, что отмечен пункт «Git Bash Here» - он добавит удобное контекстное меню для запуска терминала Git через ПКМ.

Первичная конфигурация

Откройте терминал (Git Bash на Windows, стандартный терминал на macOS/Linux) и выполните две обязательные команды: 

git config --global user.name "Ваше Имя или Никнейм"
            git config --global user.email "ваш.email@example.com"

Эти данные будут прикрепляться к каждому коммиту. Можно также настроить редактор для сообщений коммитов (по умолчанию используется Vim) и другие параметры: 

git config --global core.editor "code --wait"   # использовать VS Code
            git config --global init.defaultBranch main      # установить имя основной ветки

Проверить текущие настройки можно командой git config --list.

Регистрация на GitHub пошагово

Шаг 1: Создание аккаунта

Перейдите на github.com и нажмите кнопку Sign up в правом верхнем углу.

  1. Введите желаемый Username - он станет частью URL вашего профиля и репозиториев. Выбирайте короткое и запоминающееся имя, которое будет отражать вас как разработчика.
  2. Укажите действующий Email - на него придёт письмо для подтверждения. Рекомендуется использовать почту, которую вы не потеряете со временем.
  3. Придумайте надёжный пароль. Используйте комбинацию букв, цифр и спецсимволов.
  4. Пройдите простую проверку на робота (Puzzle) и нажмите Create account.

Шаг 2: Подтверждение email

На указанную почту придёт письмо от GitHub с кодом подтверждения. Введите этот код в открывшейся форме. Если письмо не приходит, проверьте папку «Спам» или запросите повторную отправку.

Шаг 3: Настройка профиля

GitHub предложит несколько вопросов о вашем опыте и целях использования. Можно пропустить этот шаг, нажав Skip personalization. После этого вы попадёте на главную страницу своего аккаунта.

Шаг 4: Добавление SSH-ключа (рекомендуется)

Для безопасного взаимодействия с GitHub без постоянного ввода пароля настройте SSH-ключ. Откройте терминал и выполните: 

ssh-keygen -t ed25519 -C "ваш.email@example.com"

Нажмите Enter, чтобы сохранить ключ в папке по умолчанию. Затем скопируйте содержимое публичного ключа: 

cat ~/.ssh/id_ed25519.pub

На GitHub зайдите в Settings → SSH and GPG keys, нажмите New SSH key, вставьте скопированный текст и сохраните. Теперь вы можете клонировать и пушить репозитории по SSH.

Основные понятия Git

  • Репозиторий - папка проекта, в которой Git отслеживает изменения. Может быть локальным (на вашем компьютере) и удалённым (на GitHub).
  • Коммит - снимок состояния файлов в определённый момент времени. Каждый коммит имеет уникальный хеш, автора, дату и сообщение.
  • Индекс (staging area) - промежуточная область, куда добавляются файлы перед коммитом. Это позволяет гибко формировать коммит из части изменений.
  • Ветка - отдельная линия разработки. Позволяет работать над новыми фичами, не затрагивая основную ветку (обычно main или master).
  • Удалённый репозиторий (remote) - ссылка на репозиторий, расположенный на GitHub или другом сервере. Обычно называется origin.

Создание первого локального репозитория

Предположим, у вас есть папка с проектом. Откройте терминал в этой папке и выполните: 

git init

Появится скрытая директория .git - здесь Git хранит всю историю и метаданные.

Добавим все файлы в индекс и создадим первый коммит: 

git add .
            git commit -m "Первоначальная версия проекта"

Флаг -m позволяет указать сообщение коммита прямо в командной строке. Если опустить флаг, откроется текстовый редактор для написания более развёрнутого сообщения.

Просмотр истории и состояния

Посмотреть, какие файлы изменены и что находится в индексе: 

git status

Вывести историю коммитов в кратком виде: 

git log --oneline

Для детального просмотра с информацией об авторе и дате: 

git log

Связь с удалённым репозиторием на GitHub

Создание репозитория на GitHub

  1. Нажмите на значок «+» в правом верхнем углу GitHub и выберите New repository.
  2. Введите имя репозитория. Оно должно быть уникальным в пределах вашего аккаунта.
  3. Выберите видимость: Public (виден всем) или Private (доступен только вам и приглашённым пользователям).
  4. Не отмечайте галочки «Initialize this repository with a README», «.gitignore» или «license», если вы уже создали репозиторий локально - иначе возникнет конфликт.
  5. Нажмите Create repository.

Привязка локального репозитория к удалённому

Скопируйте URL вашего репозитория (HTTPS или SSH) и выполните в терминале: 

git remote add origin https://github.com/ваш_логин/имя_репозитория.git

Проверьте, что удалённый репозиторий добавился: 

git remote -v

Отправьте коммиты на GitHub (первый пуш потребует указать ветку и настроить отслеживание): 

git branch -M main
            git push -u origin main

Флаг -u связывает локальную ветку main с удалённой origin/main, и в дальнейшем достаточно будет писать просто git push.

Клонирование существующего репозитория

Чтобы скачать чужой проект или свой с другого компьютера, используйте команду clone: 

git clone https://github.com/автор/проект.git

После клонирования Git автоматически создаст папку с именем репозитория и привяжет удалённый репозиторий origin.

Работа с ветками

Создание и переключение веток

git branch feature/awesome-idea    # создать ветку
            git checkout feature/awesome-idea   # переключиться на неё

Или одной командой: 

git checkout -b feature/awesome-idea

Посмотреть список всех веток: 

git branch -a

Слияние веток

После завершения работы над фичей нужно слить изменения в основную ветку. Сначала переключитесь на main: 

git checkout main

Затем выполните слияние: 

git merge feature/awesome-idea

Если конфликтов нет, Git автоматически создаст коммит слияния. Если конфликты есть, их нужно разрешить вручную (об этом ниже).

Удаление ветки

После успешного слияния ветку можно удалить: 

git branch -d feature/awesome-idea

Разрешение конфликтов слияния

Конфликт возникает, когда две ветки изменили одни и те же строки в одном файле. Git помечает конфликтующие участки специальными маркерами: 

<<<<<<< HEAD
            ваша версия (текущая ветка)
            =======
            версия из сливаемой ветки
            >>>>>>> feature/awesome-idea

Чтобы разрешить конфликт, отредактируйте файл, оставив нужный вариант, и удалите маркеры. Затем добавьте исправленный файл в индекс и завершите слияние: 

git add имя_файла
            git commit -m "Разрешён конфликт в файле X"

Получение и отправка изменений

Скачать свежие изменения с удалённого репозитория

git pull origin main

Команда pull - это сокращение для git fetch (получение изменений) и git merge (слияние с текущей веткой). Если вы уверены, что локальных изменений нет, можно использовать pull --rebase для более чистой истории.

Отправить свои коммиты

git push origin main

Если ветка ещё не существует на удалённом сервере, используйте: 

git push -u origin имя_ветки

Дополнительные полезные команды

Просмотр изменений

Показать разницу между рабочим каталогом и индексом: 

git diff

Показать разницу между индексом и последним коммитом: 

git diff --staged

Отмена изменений

Убрать файл из индекса, но оставить изменения в рабочем каталоге: 

git reset HEAD имя_файла

Полностью отменить изменения в файле (вернуть к состоянию последнего коммита): 

git checkout -- имя_файла

Откатить последний коммит, сохранив изменения в рабочем каталоге: 

git reset --soft HEAD~1

Удалить последний коммит и все его изменения (осторожно!): 

git reset --hard HEAD~1

Спрятать незакоммиченные изменения

git stash

Восстановить спрятанные изменения: 

git stash pop

Просмотр удалённых веток

git branch -r

Рекомендации по стилю коммитов

Хорошие сообщения коммитов экономят время вам и вашей команде. Придерживайтесь простых правил:

  • Используйте повелительное наклонение: «Добавить», «Исправить», «Обновить».
  • Ограничивайте первую строку 50 символами - это заголовок.
  • После заголовка оставляйте пустую строку, затем подробное описание (опционально).
  • Указывайте в сообщении, что именно сделано и, возможно, почему.

Пример хорошего коммита: 

Добавить валидацию email в форме регистрации

            Поле email теперь проверяется на корректность формата
            с использованием регулярного выражения. Ошибка отображается
            под полем ввода и блокирует отправку формы.

Пример рабочего процесса (Feature Branch Workflow)

  1. Создайте новую ветку от main: git checkout -b feature/name.
  2. Вносите изменения, регулярно коммитьте.
  3. Периодически подтягивайте изменения из main, чтобы избежать больших конфликтов: git pull origin main.
  4. Когда фича готова, отправьте ветку на GitHub: git push -u origin feature/name.
  5. На GitHub создайте Pull Request из вашей ветки в main.
  6. После ревью и одобрения выполните слияние через интерфейс GitHub.
  7. Удалите ветку локально и на удалённом сервере.

Что дальше?

Git - глубокая система, и этот гайд охватывает лишь основы, необходимые для комфортного старта. Рекомендуем изучить следующие темы:

  • Интерактивный rebase - для очистки истории коммитов.
  • Cherry-pick - перенос отдельных коммитов между ветками.
  • Теги (tags) - для отметки релизных версий.
  • Git Hooks - автоматизация проверок перед коммитом или пушем.
  • GitHub Actions - CI/CD прямо в репозитории.

Отличный интерактивный тренажёр для закрепления навыков: Learn Git Branching. А официальная документация Git всегда доступна по команде git help <команда> или на git-scm.com/doc.

Заключение

Использование Git и GitHub - обязательный навык для любого современного разработчика. Даже если вы работаете в одиночку, система контроля версий сэкономит нервы при случайном удалении файла или неудачном эксперименте. Начните применять Git уже сегодня, и через неделю вы будете удивляться, как раньше обходились без него.
